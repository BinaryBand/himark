## Performance

- [x] ~~**Deferred capture materialization**~~ — **done (general engine win).** The match loop built each capture's text slice and `reps[:k]` trim on *every* backtracking count it tried, so a long `[..]`/`[1..]` run that backs off toward a constraining continuation cost O(L²). Captures now defer both (text re-derived from the absolute span on demand by the rare back-reference that reads one mid-match, trimmed once at `_finalize`). Helps any backtracking-heavy pattern: dedup **n=80 6.9→2.9s, n=120 17.1→5.2s** (speedup grows with n — order cut, not a constant), and field-length scaling went from quadratic to ~linear (**41→5.8s** at 4× field width). Identical output; all tests pass.

- [x] ~~**Look-ahead ladder**~~ — **tested, harmful (do not pursue).** A geometric ladder of QUICK windows (`[..300]/[..3000]/[..30000]`, etc.) ran **1.4–3× slower** than the single `[..3000]` pass on 80/120-row slices (identical output): each extra bounded sweep re-scans every partnerless singleton, and that overhead outweighs the unbounded passes it saves. The single tuned window stays.

- [x] ~~**Radix sort + linear merge** / **Key-prefix blocking**~~ — **investigated, blocked in pure HMK (needs a new primitive).** Both reduce to *grouping equal keys adjacent*, i.e. a sort. But HMK relocates only by text-rewrite — there is no O(1) bucket/pointer placement — so **every HMK sort is O(n²)** (bubble_sort measured clean O(n²): 100→0.72s, 200→3.04s, 400→12.53s). Sorting dedup's ~1718 keyed lines extrapolates to **~230s — about the same as the whole current MATCH (~266s)** — so sort-then-merge cannot beat the lazy equality-grouping the unbounded sweep already does, and blocking-by-tag gives nothing because the `{$0}` key compare already fails fast on a first-char mismatch (the cost is *visiting* O(n) line-starts, not compare depth, and that needs contiguous buckets = a sort). The real lever is a **language/engine primitive**: a built-in sort/group, a hash-join seam, or an O(1) relocation op — a design decision, not a `dedup.hmk` change. Until then, the realistic Python win was the deferred-capture fix above (~3×); the native backend stays the answer for the full file.

- [x] ~~**Key fingerprinting**~~ — **investigated, not worth it.** The idea (hash the key short, keep the full key for a verify step) is self-defeating: a fingerprint shorter than the key can't be injective, so to avoid *wrongly merging two different episodes* you must keep the full key — making each line *longer*, not shorter. The scan cost is byte-traversal (post-deferral, ~linear in field length), so more bytes = slower, not faster. Per-compare cost was never the bottleneck anyway — the `{$0}` compare already fails at char 1 for non-partners; the cost is the O(n) line-starts *visited* per Y-line, which fingerprinting doesn't reduce, and it leaves the O(n²) asymptotic untouched. (It also needs a string-hash primitive `b256`/`uint`/`pad` don't provide.)

- [x] ~~**Drop redundant P-line key/title duplication**~~ — **done, but smaller than hoped (~5%).** A P line stored `<title>¤P\t<title>` with key == title, so the payload was a pure copy; it now stores `<title>¤P\t` and the merge/cleanup read the P column straight from the shared key (`{{$0}}`). Identical output, all tests pass, and the script is cleaner (real redundancy removed). The speedup was only ~5% (n=120: 5.2→4.9s), not the estimated ~1.3×: post-deferral the cost is dominated by the *count* of match attempts / line-starts visited, not raw byte traversal — and this removes bytes, not lines.
